Building a local AI stack worth using

Six months of running modern AI tooling end-to-end on a personal machine — what earned its keep, what didn't, and what surprised me.

Updated April 2026 · 14 min read · self-initiated

I started this in late 2025 with a question that wouldn't go away: how much of the AI work that's actually interesting could I do on a single laptop, without standing up cloud infrastructure or paying API bills that I'd notice on my credit card statement? The honest answer turned out to be more than I expected — but with a different shape than I assumed going in.

This is a writeup of the stack I converged on, the trade-offs that surprised me, and the things I'd do differently. It's not a tutorial; it's the version of the story I'd tell a friend over coffee, with enough specifics to be useful and enough opinions to be interesting.

Why local at all

There's a version of this story where "local" is about sovereignty or privacy, and I think that version is true but not interesting. The interesting version is about iteration friction — how much faster you actually move when "try this model on this data" is two terminal commands instead of a setup meeting with yourself.

The latency story is real but it isn't the main thing. The main thing is that you stop noticing setup cost, which means you experiment more, which means you learn faster. Privacy is the third reason, and a real one for some of what I've been doing, but it's downstream of the first two.

The stack, in order of importance

ollama Local LLM runner. The thing I open most. DAILY
claude api For the work local can't yet do well. DAILY
duckdb Anything tabular touches it first. DAILY
lm studio Model exploration and quick A/B. WEEKLY
qdrant Local vector store for embeddings work. WEEKLY
llamafile Shipping a model as a single binary. OCCASIONAL
daily driver regular use specific cases

The ordering matters. ollama at the top isn't because it's the most powerful piece — it isn't — but because it's the piece I reach for without thinking, which is the piece doing the most real work. The Claude API sits next to it because there's a clean line between "local can do this well" and "local can't yet," and ignoring that line wastes a lot of time.

What surprised me

Three things I didn't expect going in.

First, embeddings turned out to be more useful than chat for the workflows that stuck. Chat is the obvious surface, but a small embedding model running locally over my own notes and bookmarks is the thing I actually open every day. The chat output is forgettable; the search-over-my-own-stuff isn't.

The most useful upgrade wasn't a new model. It was getting the inner loop down to under five seconds.

Second, modern quantization is shockingly good. The gap between an 8-bit and 4-bit model on tasks I care about is much smaller than I expected, and a 4-bit model that fits in memory and runs at conversational speed is more useful than an 8-bit one that doesn't.

Third — and this is the one I should have seen coming — the orchestration layer is the actual work. The model is a commodity-ish input. The pipeline you build around it is the thing that's yours. Most of my time has gone into the glue, not the model choice.

# the inner loop that earned its keep
for chunk in stream_chunks(local_model, prompt):
    annotate(chunk)
    if confident_enough(chunk):
        commit(chunk)
    else:
        escalate(chunk, fallback="claude")

The escalation pattern at the bottom is doing a lot of work in this codebase. Cheap-and-local for the easy cases, frontier-API for the hard ones, with a calibrated confidence check between them. It's not novel — it's the obvious shape — but getting the threshold right took longer than I'd like to admit.

What I'd do differently

If I started over today, I'd spend less time on model selection and more time on evaluation. I underweighted the importance of having a small fixed set of tasks I cared about and a way to score outputs against them; without that, you end up vibing on which model is "better," which is fine for a hobby and bad for anything you want to ship.

I'd also pick the orchestration framework earlier. I rolled my own glue for too long, partly out of stubbornness and partly because I wanted to understand the shape of the problem first. That second reason is defensible; the first one cost me weeks.

The Boston civic-data project I'm working on next is, in part, a forcing function for that — it's a real workload, with a real evaluation set, on data I actually care about. If you want to know what I'd actually do differently, watch that project.
