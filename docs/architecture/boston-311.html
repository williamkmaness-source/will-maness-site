Boston 311 Tracker

Department backlog equity dashboard — which city departments are falling behind and is it getting worse?

Active

Overview

The Boston 311 dashboard answers one operational question: which city departments are showing signs of stress before a council office gets there first? Boston publishes every 311 service request (potholes, missed trash, noise complaints) as open data. The dataset updates continuously and goes back years.

The dashboard tracks weekly open vs. closed ticket counts and median closure time for each of the nine city departments that handle 311 work. When both metrics worsen for two or more consecutive weeks, the department is flagged — giving an operations manager a clear signal before a backlog becomes a crisis.

Framing: The dashboard is framed as operational reallocation intelligence, not staffing analytics. It surfaces where resources need to shift, not headcount decisions. This framing matters for how headers and copy are written.

Flagging Logic

ConditionFlagNotes
Backlog growing (opened > closed) AND median closure time increasing — 1 consecutive weekYellow (watch)Single stressed week; may be noise
Same dual condition — 2+ consecutive weeksRed (flagged)Sustained deterioration; escalation signal
Either condition improvesReset to greenCounter resets to zero on any calm week

Methodology choices: median closure time (not mean, to resist outlier distortion). Both conditions must hold simultaneously — a single bad metric is noise. Two-week minimum for red.

Agent Notes

⚠️ Dashboard framing is "operational reallocation," not "staffing." The distinction matters for copy. Headers and descriptions must not suggest headcount decisions. The dashboard surfaces where resources need to shift, not how many people to hire or fire.
⚠️ The backfill is complete. case_events has 95K rows going back to 2024 (backfilled 2026-05-18). Do not attempt to re-run the backfill script — it is a one-time operation already completed.
⚠️ pipeline-311.ts must upsert pipeline_runs on every run. The pipeline dashboard reads from this table for health status. If a pipeline run skips the upsert, the health card shows "Unknown." This was a historical bug (fixed 2026-05-24).
⚠️ API route needs force-dynamic. The /api/311-departments route hits Neon Postgres and must not be statically prerendered. Add export const dynamic = 'force-dynamic'.
💡 Median, not mean. A single case open for 2+ years would distort the mean dramatically. All closure time metrics use median. Do not change to mean.
💡 Forecasting (#58) is gated on data accumulation. Do not start the forecasting issue until at least 8 weeks of post-backfill pipeline data exists. The model needs ~125 weeks of weekly data to fit meaningful seasonal patterns — the backfill provides historical data, but model validation requires recent weeks.
