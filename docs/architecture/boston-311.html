Overview

An operational reallocation dashboard built on Boston's public 311 service request data. The dashboard answers: which neighborhoods wait longest for city services, by request type, and what does that gap cost in staff-hours? Framed as a staffing tool for department heads, not a complaint tracker.

Data is sourced from the Open Boston CKAN portal (Analyze Boston) — public, no auth required. A nightly pipeline fetches new cases and upserts them into Neon Postgres. The dashboard reads from Neon and renders charts with Recharts.

Historical backfill covers 2024-01-01 to present (95K+ rows as of 2026-05-18). New rows accumulate nightly.

Agent Notes

Neon parameterized queries: Use sql.query(query, params) for the batch upsert — NOT sql(query, params). Neon v1.1 dropped the old convention. Tagged template literals still work for simple queries.
CKAN SQL injection guard: Date parameters are validated against /^\d{4}-\d{2}-\d{2}$/ before string interpolation. CKAN's SQL endpoint does not support parameterized queries, so this regex is the injection boundary. Don't relax it.
  • Dashboard framing: The project is an operational reallocation dashboard, not a staffing complaint tracker. Copy and UI choices should reinforce this framing.
  • YEAR_RESOURCES: Three resource IDs in boston-311.ts — one per calendar year (2024, 2025, 2026). A new resource ID must be added when Boston publishes the 2027 dataset.
  • Backfill is safe to re-run: All upserts use ON CONFLICT (case_id) DO UPDATE. Re-running the backfill script won't duplicate rows.
  • Migrations before deploying schema changes: Run pnpm tsx scripts/migrate-311-departments.ts against prod Neon before any new DB-backed feature goes live.
  • force-dynamic: Any page or route that reads from Neon must export const dynamic = "force-dynamic".
