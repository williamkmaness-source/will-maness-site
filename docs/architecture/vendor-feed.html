Vendor Intelligence Feed

Daily pipeline tracking feature launches, pricing changes, partnerships, and architectural shifts from 15 data integration companies

Active

Overview

An automated pipeline that monitors 15 data integration companies (Fivetran, dbt Labs, Airbyte, etc.) for competitive intelligence signals. Every morning, a GitHub Actions job scrapes blog posts and GitHub release notes, sends raw content to Claude Haiku for structured extraction, and upserts the results into Neon Postgres. A Next.js page renders the feed — filterable by company and entity type, with filter state reflected in the URL.

Pipeline runs in three stages: (1) Scrape — fetch pages, hash content (SHA-256), skip unchanged pages; (2) Extract — Claude Haiku via tool_use extracts all four entity types in a single call; (3) Upsert — write entities to typed tables, deduplicate by source URL + content hash. A weekly retry cron resets failed pages to pending for re-processing.

Agent Notes

⚠️ Do not start #68 (embeddings) until the MVP bar is met. The issue is explicitly marked Post-MVP. Prerequisites: 30+ entities, 5+ companies, 3+ successful pipeline runs. Picking it up early wastes effort on infrastructure that has no frontend surface yet.
⚠️ Claude Haiku is the correct model here. Haiku is used for cost efficiency — extraction runs daily across 15 companies. Do not switch to Sonnet or Opus without understanding the cost impact of daily high-volume extraction.
⚠️ Company list is derived from actual data for filter rendering. The filter dropdown only shows companies with at least one extracted entity. Empty companies do not appear as filter options. This means freshly added companies won't appear until their first extraction completes.
💡 Deduplication is content-hash based, not date-based. If a company re-publishes old content with the same URL but changed text, a new extraction runs. If the URL and content hash both match, the page is skipped entirely — even if the extracted entity record changed.
💡 The retry cron resets status='failed' to 'pending' weekly. Transient scrape failures (timeouts, 5xx) are automatically retried on the next weekly cycle. Do not manually reset statuses unless debugging a specific extraction issue.
