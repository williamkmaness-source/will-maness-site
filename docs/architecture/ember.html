EmberBrief

Wildfire intelligence dashboard — Lake Tahoe Basin fire detection, weather scoring, and AI situation briefings

Active

Overview

EmberBrief translates live fire and weather feeds into plain-language situation briefings, automatically. NASA's FIRMS satellite network detects fire heat signatures within minutes; NOAA weather stations report conditions in real time. The bottleneck is synthesis — EmberBrief removes that step.

A Vercel cron job runs daily: fetch active fire detections from NASA FIRMS NRT feed → fetch current weather from the nearest NOAA Synoptic station → cluster detections within 2km into discrete fire events → score each cluster on a transparent risk index → for Action-tier events, generate a plain-language situation briefing via the Claude API.

Geographic scope: Lake Tahoe Basin bounding box. Weather station: LIB33 ("Tahoe-7300 Sugar Pine Point"). The risk score is a weighted composite of FRP intensity (40%), weather danger (40%), and trend over prior 3 hours (20%). Tiers: Monitor (<40), Watch (40–69), Action (≥70). Claude is called only for Action-tier clusters — keeping AI costs proportional to actual risk.

Architecture Diagram

Daily pipeline (Vercel Cron — 09:00 UTC):

NASA FIRMS NRT ──HTTP──→ ┐
                          ├── ember-ingest route handler
NOAA Synoptic v2 ──HTTP──→ ┘
                          │
                          ├── 1. Parse fire detections (VIIRS CSV)
                          ├── 2. Cluster within 2km radius
                          ├── 3. Fetch 3h FRP history from Neon (trend calc)
                          ├── 4. Compute risk score (FRP 40%, weather 40%, trend 20%)
                          ├── 5. Assign tier: Monitor / Watch / Action
                          ├── 6. Evaluate Red Flag (RH <15% AND wind >25mph)
                          ├── 7. [Action only] Call Claude Haiku → briefing text
                          └── 8. Upsert to Neon Postgres
                               ├── ember_fire_clusters (one row per cluster per day)
                               └── ember_county_conditions (weather snapshot)
                                          ↓
                             /api/ember-data (Route Handler, force-dynamic)
                                          ↓
                                /ember (Dashboard page, SSR)

Agent Notes

⚠️ SYNOPTIC_API_KEY must be a token, not the API key. Error "Be sure to use a token generated from your API Key" means the wrong value is stored. Generate tokens at developers.synopticdata.com. The raw API key silently fails.
⚠️ ANTHROPIC_API_KEY is Production-only. The cron is designed to skip the briefing step gracefully when the key is absent — it logs briefed: 0 rather than throwing. Do not add this key to preview/development environments.
⚠️ Both FIRMS and Synoptic keys were historically set to "False" as placeholders. Silent pipeline failures ran for the entire lifetime of the project before being discovered (2026-05-26). Always verify env var values are real keys, not placeholders.
⚠️ Red Flag is county-level, not cluster-level. Red Flag (RH <15% AND wind >25mph) is surfaced as a separate county flag — it is NOT embedded in the cluster risk score. This is intentional: Red Flag conditions matter even with no active fire.
💡 Claude is only called for Action-tier events (score ≥ 70). Most daily runs produce zero Claude calls. Do not change this threshold without understanding cost implications.
💡 The briefing prompt is fully deterministic. Claude cannot infer beyond what is supplied — the prompt is constructed from exact cluster metrics. Three output sections mirror trained analyst output: Current Situation, Weather Context, 6-Hour Outlook.
💡 Synoptic valid variable names: wind_speed, wind_direction, wind_gust, relative_humidity, air_temp, precip_accum. Parsed from obs keys: wind_gust_value_1, precip_accum_value_1. The variable precip_accum_24h is NOT valid.
