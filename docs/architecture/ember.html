Overview

EmberBrief translates live fire and weather data into plain-language situation briefings, automatically. When a wildfire ignites, NASA satellites detect the heat signature within minutes and NOAA weather stations report conditions in real time. The bottleneck is synthesis — EmberBrief removes that step.

The live dashboard at /ember shows current fire activity and conditions for the Lake Tahoe Basin, scored and briefed automatically on a daily cadence. A Vercel cron job runs every day at 09:00 UTC, orchestrating: FIRMS ingest → clustering → weather fetch → risk scoring → AI briefing (Action tier only) → Neon upsert.

The AI layer boundary is deliberate: rules handle everything with a ground-truth answer (clustering geometry, risk thresholds, Red Flag flags). Claude handles exactly one thing — synthesizing structured data fields into a briefing paragraph a non-analyst can act on.

Pipeline Detail

Step 1 — FIRMS Detection Fetch

Calls https://firms.modaps.eosdis.nasa.gov/api/area/csv/{apiKey}/VIIRS_SNPP_NRT/{area}/1.

  • Bounding box: west=-120.5, south=38.7, east=-119.5, north=39.4 (Lake Tahoe Basin + surrounding foothills)
  • Day range: 1 (last 24 hours)
  • Confidence filter: drops l (low confidence) — false alarms damage trust more than missed fires
  • Timeout: 20 seconds

Step 2 — Clustering

Groups VIIRS pixels within 2 km of each other using a greedy spatial grouping algorithm (cluster-engine.ts). Each cluster's centroid, total FRP, and detection count are computed from the member pixels.

Step 3 — Weather Fetch

For each cluster: calls Synoptic API for the nearest station within 30 miles. Extracts: wind speed (mph), wind direction (deg → cardinal), wind gust (mph), relative humidity (%), temperature (°F), 24h precip (in). Stored as JSONB in ember_fire_clusters.weather.

Step 4 — Risk Scoring

See Scoring Model below. Result stored as risk_score (0–100) and tier (Monitor/Watch/Action).

Step 5 — AI Briefing (Action tier only)

Constructs a BriefingInput from the cluster's metrics and calls generateBriefing(anthropicClient, input). Skipped entirely if ANTHROPIC_API_KEY is not set. Output is a structured JSON object with three sections: current situation, weather context, 6-hour outlook. Stored in briefing column.

Step 6 — County Conditions

Fetches weather for the Lake Tahoe Basin centroid (lat=39.05, lng=-120.05) and upserts a single row in ember_county_conditions keyed on county='Lake Tahoe Basin'. Includes a Red Flag evaluation: RH < 15% AND wind > 25 mph.

Step 7 — Pipeline Health

Upserts pipeline_runs with pipeline='ember', status='success', and the cluster count. On failure, sets status='failed' and writes the error message.

Risk Scoring Model

Risk score is a weighted composite of three components, each normalized 0–100:

ComponentWeightFormula
FRP Intensity40%Linear ramp from FRP_LOW=10 MW (→ 0) to FRP_HIGH=500 MW (→ 100)
Weather Danger40%Average of RH score (100 − humidity%) and wind score (wind_mph / 50 × 100). Red Flag (RH<15% AND wind>25 mph) overrides to 100. Null variables use 50 (neutral).
Trend20%FRP change vs prior cluster within 10 km. 50=neutral, clamped 0–100. Null prior = 50.

Tier thresholds:

  • Monitor — score < 40
  • Watch — score 40–69
  • Action — score ≥ 70 (triggers AI briefing)
Red Flag conditions (county-level RH < 15% AND wind > 25 mph) are surfaced as a separate flag on the dashboard, independent of cluster risk scores. They matter regardless of active fire.

Agent Notes

Full table DELETE on every run: ember_fire_clusters is truncated and re-inserted on each cron run. Never rely on row IDs persisting across runs. The prior run's rows are snapshotted in memory before DELETE for trend scoring — but they don't stay in the DB.
ANTHROPIC_API_KEY absent = silent skip: In preview and local dev, the briefing pass is skipped without error. This is intentional. If you need to test briefing generation locally, add the key to .env.local.
Neon API convention: The ingest route uses tagged template literals (sql\`...\`). For any parameterized queries you add, use sql.query(query, params) — the old sql(query, params) was dropped in Neon v1.1.
  • Scoring constants are named: Never use magic numbers. All thresholds are exported constants in ember-scoring.ts — FRP_WEIGHT, WATCH_THRESHOLD, MAX_DANGER_WIND_MPH, etc.
  • Confidence filtering: Low-confidence VIIRS detections ('l') are dropped before clustering. This is a product decision — false alarms damage trust more than missed fires. Don't change without discussing.
  • Red Flag is county-level, not cluster-level: The Red Flag boolean in ember_county_conditions reflects basin-wide conditions. Cluster weather stores raw wind/RH but the Red Flag badge on the dashboard comes from the county table.
  • FIRMS bounding box: Defined as LAKE_TAHOE_BASIN_BBOX in firms-client.ts. Expanding coverage means changing this constant and verifying Synoptic station coverage for the new area.
  • Weather fetch is parallel: Each cluster's weather is fetched concurrently via Promise.all. Individual failures are logged but don't fail the whole run.
  • pipeline_runs is required: The pipeline dashboard shows "Unknown" if ember-ingest doesn't upsert pipeline_runs. Never remove the upsert step.
