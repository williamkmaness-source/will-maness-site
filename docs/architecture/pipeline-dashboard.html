Overview

A health dashboard that shows the last run status of all active data pipelines powering the site. Each pipeline that writes to Neon gets one card: status (success / failed / unknown), last success timestamp, last attempt timestamp, records processed, and error message on failure.

The dashboard is a portfolio transparency tool — it proves the pipelines are actually running and gives visitors confidence that the live features show real data.

Architecture Diagram

  Each pipeline (ember-ingest, 311 nightly) upserts pipeline_runs after each run:

    INSERT INTO pipeline_runs (pipeline, status, last_success_at, last_attempt_at, record_count, error)
    ON CONFLICT (pipeline) DO UPDATE SET ...

  Dashboard reads:

  Browser  ──→  usePipelineStatus hook (polls /api/pipeline-status every 60s)
                        │
                        └─ /api/pipeline-status  ──→  Neon pipeline_runs  ──→  JSON
                                                        SELECT * FROM pipeline_runs
                                                        ORDER BY pipeline

  PipelineDashboard renders one PipelineCard per row returned.

Agent Notes

If a pipeline shows "Unknown": the pipeline is not upserting pipeline_runs after each run. Every pipeline must write to this table — even on failure — or the dashboard card will show stale / unknown status forever.
  • Adding a new pipeline: The pipeline key must be unique in pipeline_runs. Pick a short, lowercase slug (e.g. 'vendor-feed'). The dashboard automatically picks up new rows — no frontend change needed.
  • force-dynamic: /api/pipeline-status must export dynamic = "force-dynamic" since it reads from Neon on every request.
  • Incident history: A stale branch in the Claude OS repo was incorrectly set as default branch, causing all 3 pipelines (including chess) to fail daily. Fixed 2026-05-24. If pipelines show phantom failures, check GitHub Actions branch configuration first.
