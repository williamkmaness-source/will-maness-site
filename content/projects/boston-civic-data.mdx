Boston publishes every 311 service request — every pothole report, every missed trash pickup, every noise complaint — as open data. The dataset goes back to 2011 and updates continuously. What it doesn't do is tell you how fast the city is responding to those requests.

> Can the city of Boston keep up with the volume of requests that are coming in every month? Is the rate of closing these requests meaningfully different across neighborhoods? The data is public, but not clearly summarized - until now.

That's the project. The [Boston 311 Equity Tracker](/work/boston-civic-data) computes a series of visualizations representing how expeditiously the city is closing down 311 tickets. It ranks every neighborhood by the net change in requests and the speed to close those requests with filters by request type to specify certain activities.

## How it works

The city's data lives on the Analyze Boston CKAN platform, split into per-year CSV resources. A GitHub Actions job runs every night, queries the API for new cases, and upserts them into a `case_events` table in Neon Postgres — one row per 311 case, keyed on the city's case ID. The Next.js route handler then computes all metrics live using Postgres window functions (`PERCENTILE_CONT`), reading directly from `case_events` on every request.

A few deliberate choices in the methodology:

- **Median, not mean.** A single outlier case open for two years drags the mean. Median gives a better read on what a typical resident actually experiences — and `PERCENTILE_CONT` computes it exactly over the raw case data, with no preaggregation artifacts.
- **`reason` field, not `type`.** More stable across years and produces cleaner groupings for filtering.
- **Per-request retry with backoff.** The nightly pipeline retries each CKAN call up to three times with exponential backoff before failing — CKAN can be slow, and a transient hiccup shouldn't abort the run.

## What's next

The `case_events` table now holds ~95,000 historical cases going back to 2024, with the nightly pipeline keeping it current. The next phase is adding department and SLA context — which city department handles each request type, and what the official response target is. That data is already being tracked in a `request_type_meta` table; wiring it into the UI will let the tracker answer a harder question: is a neighborhood's slow response time a city-wide problem for that request type, or is it specific to that neighborhood?
